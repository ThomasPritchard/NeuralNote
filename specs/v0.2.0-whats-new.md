# NeuralNote 0.2.0 What's new surfaces

Status: proposed for maintainer approval on 15 July 2026.

This specification adds a single, app-owned v0.2.0 changelog to Settings and
to the first launch of the installed version. It does not authorise a commit,
push, merge, tag, release, or deployment.

## Outcome

NeuralNote 0.2.0 has one authoritative set of release notes. Opening Settings
normally lands on those notes instead of an empty or task-specific canvas. The
same content appears in a `What's new in NeuralNote 0.2.0` modal the first time
the installed app is opened after the version changes.

The startup modal is application-level. It can appear before a vault is open,
and its acknowledgement applies to the installation rather than to one vault.
It appears once for each app version, not once per launch or once per vault.

## Chosen design

Use a static, typed release-notes module bundled with the app. A structured
release record contains the version, title, introduction, and ordered groups of
plain-text items. A shared presentational component renders that record in both
surfaces.

This is preferred to fetching release notes from GitHub because the first-run
journey must work offline and must not add a network or remote-content trust
boundary. It is also preferred to maintaining separate JSX copies because the
Settings page and startup modal must not drift.

The current release record is selected from the app's build version. A missing
record is explicit in development and does not render stale notes under the
wrong version.

### Settings

- Add `What's new` as the first Settings section and the default section for a
  normal Settings open.
- Keep direct routes such as `Manage local models` opening their existing
  target section.
- Keep General, Appearance, Templates, AI, Skills, and About unchanged and
  reachable from the existing navigation.
- Render the release notes as a semantic, scrollable article headed
  `What's new in NeuralNote 0.2.0`.

### First launch after an update

- Mount the coordinator above the welcome/workspace router so it works with or
  without an open vault.
- Show the modal when the stored last-seen version is absent or differs from
  the current version.
- Persist the current version only when the user dismisses the modal. If the
  app exits before dismissal, show it again next time.
- Store the acknowledgement in the existing application preferences file, not
  in vault data. Add a nullable `lastSeenWhatsNewVersion` preference with a
  backward-compatible default for existing preference files.
- If acknowledgement persistence fails, close the modal, surface the existing
  explicit settings-save error, and allow it to appear again next launch. Do
  not claim it was saved.
- Do not show the startup modal again merely because Settings is opened.

## NeuralNote 0.2.0 changelog content

The shipped copy is grounded in `main..feature/neuralnote-0.2.0`, the approved
0.2.0 interaction specification, and the uncommitted source-native editor work
in this worktree. It must not claim roadmap features such as complete
full-source capture, chunk-level citations, sync, or mobile support.

### Source-native editing

- Markdown is now editable in one live-preview surface, with supported syntax
  styled in place while the source remains authoritative.
- Headings, including a note's leading H1 title, remain part of the document
  and can be edited directly.
- Standard links and resolved Obsidian wikilinks can open their target through
  the guarded vault resolver; unresolved and unsafe links remain inert.
- `[[` completion chooses notes with duplicate-path disambiguation and leaves
  the caret inside the link so heading or block fragments and display text can
  be added directly.
- Headings, emphasis, lists, tasks, blockquotes, code, tables, links, callouts,
  block IDs, and other conservative Obsidian treatments render in place.
- Unsupported, malformed, plugin-owned, Dataview, HTML, MDX, and JSX syntax
  remains visible and editable without executing.
- Save, dirty state, conflicts, reload, overwrite, tabs, Properties,
  backlinks, formatting actions, and source-preserving line endings continue
  through the full-document write boundary.

### Neural Assistant AI

- `Cited recall` is now `Neural Assistant AI`, retaining honest note-and-line
  citations.
- The composer shows the active model and offers a keyboard-accessible model
  picker.
- OpenRouter choices use a bounded, validated list of compatible tool-capable
  models while keeping the current selection stable when rankings change.
- Local AI users can jump from the composer to the existing model-management
  settings.
- Active responses have a `Stop response` action that keeps partial output and
  distinguishes a user stop from provider or lifecycle failures.

### Workspace and presentation

- Files and Search use symmetric pane toggles and preserve their mounted state
  while hidden.
- Sidebar layout state migrates safely and restores the selected pane and
  preferred width.
- The macOS titlebar follows native fullscreen state without disturbing other
  platforms.
- Note paths use the primary interface typeface, sidebar typography is refined,
  and the welcome screen uses the tracked NeuralNote mark.

### Reliability and release readiness

- OpenRouter catalogue and ranking traffic is bounded, validated, and kept
  behind the native keychain boundary.
- Model selection changes only the selected model and preserves the provider
  key.
- Editing keeps expected-hash conflict detection, vault containment,
  symlink/race protection, atomic writes, recoverable drafts, and the 8 MiB
  editable-note limit.
- Application, package, updater, and local upgrade tooling are aligned on
  version 0.2.0.

## Accessibility and security

- Both surfaces use semantic headings and lists. The startup surface is a
  labelled modal with focus trapping, Escape handling, a visible close action,
  focus return, and an independently scrollable body.
- Content is bundled plain text rendered through React text nodes. It cannot
  contain remote HTML, scripts, external images, or arbitrary links.
- The last-seen version is untrusted preference data. Compare it only as a
  bounded version string and never use it to select executable content.
- This feature adds no network request, vault write, native navigation, or new
  privileged command.

## Test-first delivery

Start with failing tests for:

- the normal Settings open defaulting to `What's new` while explicit section
  routes still open their target;
- both surfaces rendering the same v0.2.0 record and complete section set;
- a fresh install and a version change opening the modal before a vault exists;
- the matching stored version suppressing the modal across app remounts and
  vault changes;
- dismissal persisting 0.2.0 only after the close action;
- a failed preference write surfacing an error without a false acknowledgement;
- keyboard focus, Escape, focus return, accessible names, semantic article
  structure, and scroll containment;
- old preference files loading with a null last-seen version and corrupt
  preference recovery remaining explicit.

After focused red-green cycles, run the complete applicable frontend, Rust,
bindings, coverage, build, audit, quality, secret-scanning, UI-review, and
adversarial-review gates. Rebuild `target/dev-builds/NeuralNote-DEV.app` and
stop for hands-on verification before committing.
