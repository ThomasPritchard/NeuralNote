# NeuralNote 0.2.0 interaction release

Status: approved for implementation on 14 July 2026.

This specification defines the interaction, editing, provider-selection, and
release work for NeuralNote 0.2.0. It extends the existing product and slice
contracts. Where it is more specific, this document controls the 0.2.0 work.

## Outcome

NeuralNote 0.2.0 should feel like one coherent desktop application rather than
a collection of separate read, edit, and AI modes. Notes are editable in place,
the active AI model is visible and selectable where conversations happen, and
the window chrome and navigation remain stable as the workspace changes.

Delivery has four distinct checkpoints:

1. **Implementation ready:** the changes are verified locally, committed to a
   feature branch, pushed, and opened as a pull request. All required pull-request
   checks and a manually dispatched native E2E run for the branch are green.
2. **Merge ready:** the pull request remains open until the maintainer explicitly
   approves merging it. Merge approval authorises only the merge.
3. **Release ready:** the exact merged `main` commit has passed main CI and native
   E2E. The maintainer must then give a second, explicit approval that names the
   tag, exact commit, signing mode, and any required signing acknowledgement.
4. **Release complete:** the approved release workflow has succeeded and the
   published release, assets, updater manifest, and upgrade journey have all
   passed the post-publication checks in this specification.

The overall 0.2.0 task is complete only at the fourth checkpoint. Approval to
merge never authorises creating a tag, dispatching a release workflow,
publishing assets, or deploying the application.

## Scope

### Neural Assistant AI

- Rename the workspace feature currently labelled `Cited recall` to
  `Neural Assistant AI`.
- Keep the product promise honest: answers cite vault notes and line evidence;
  this release does not claim complete full-source capture or chunk-level
  recall.
- Replace the passive model pill with a model menu in the chat composer footer.
- Keep model names and provider metadata in the existing monospace treatment.

#### OpenRouter model menu

When OpenRouter is active, the menu must:

- Show the selected model, including a pinned entry when it is not in the
  current ranked list.
- Offer up to ten highest-ranked compatible models from the latest completed
  UTC day in OpenRouter's daily rankings. Fewer than ten is a valid result; do
  not backfill unranked catalogue entries as though they were daily-ranked.
- Join ranking records to the public model catalogue rather than trusting
  display data from a single response.
- Include only models that advertise tool support and at least 32,768 context
  tokens.
- Cache the successful result for the current UTC day and expose an explicit
  retry after a failure.
- Never change the selected model merely because rankings changed.
- Disable model changes while an AI run is active.
- Show model choices only in the ready menu. Each radio item uses the model's
  display name and selected check state; do not show ranking numbers, a
  `Current` label, model identifiers, context metadata, or a Source/OpenRouter
  footer.

The native shell owns provider requests because the API key must remain in the
OS keychain. External JSON is untrusted: requests use fixed OpenRouter origins,
bounded response bodies and timeouts, defensive schema validation, and
user-visible errors. Keys, raw provider bodies, ranking token totals, and
credentials must not appear in logs or webview IPC. The shell may return only a
bounded, validated app-owned DTO for up to ten entries containing `id`, `name`,
`contextLength`, `rank`, and `asOf`, plus the selected model when it is pinned
separately.

Selecting a model uses a dedicated command that updates only the OpenRouter
model. It must not require, return, or overwrite the provider key. Pure ranking,
filtering, validation, and joining logic belongs in `neuralnote-core`; the
Tauri shell remains a thin I/O boundary.

If ranking or catalogue retrieval fails, NeuralNote keeps the current model and
conversation usable. It must not silently substitute another model.

#### Local model menu

When local AI is active, the menu shows the selected local model and a
`Manage local models` action. That action opens the existing AI settings screen.
This release does not duplicate local-model discovery or lifecycle controls in
the composer.

### Stop response

The existing end-to-end cancellation path becomes an explicit `Stop response`
action while an AI run is active.

- A user stop leaves the partial answer visible.
- Provider streaming and any in-flight skill work stop promptly.
- The conversation shows a neutral `Stopped` state rather than a chat failure.
- Provider, vault-close, window-close, and unexpected cancellation failures
  retain their existing explicit error treatment.
- The `ChatEvent` wire contract does not change for this release. The webview
  creates a validated UUID `turnId` before starting a run and passes it to both
  `chat` and `cancel_chat_run`. The cancel command may act only when that ID
  matches the current run and returns a generated typed outcome that echoes the
  ID: `cancelled`, `already_completed`, or `not_current`. A delayed cancel can
  never target a later turn.
- The core run path carries an internal typed termination reason. It returns a
  typed user-stop outcome before emitting a terminal error, or uses a
  cancellation-aware event-sink adapter that suppresses only an error causally
  tagged with the matching user-stop reason. A user stop is never inferred from
  error text. Provider and lifecycle failures bypass this suppression and emit
  their existing `ChatEvent::Error`.

The action must be reachable by keyboard, have an accessible name, and cannot
submit a second request while stopping.

Cancellation is resolved per active turn:

- Clicking stop immediately changes the action to `Stopping`, disables it, and
  sends cancellation for that turn in the next task. A deterministic test
  provider must observe the signal within 100 ms and terminate within one
  second.
- If normal completion wins before cancellation is acknowledged, completion
  wins and the turn is not labelled stopped.
- If the typed outcome is `cancelled` while the turn is still active, mark that
  exact turn as user-stopped. Keep its partial content and show `Stopped`.
- If the typed outcome is `already_completed` or `not_current`, retain the
  normal terminal state for the identified turn. If the cancel command fails,
  surface `Couldn't stop the response` and process the next terminal event
  normally.
- Events for another turn, lifecycle cancellation, and provider failures are
  never reclassified as a user stop.

### In-place note editing (superseded)

The approved source-native live-preview direction is specified in
`specs/source-native-live-preview-editor.md`. That document supersedes this
section, including its MDXEditor compatibility matrix, rich/raw fallback,
source-range patch protocol, editor-specific tests, and non-goal against broad
Obsidian syntax editing. The remaining non-editor requirements in this release
specification continue to apply.

### Window chrome

When the macOS window enters native fullscreen, change the titlebar left-cluster
padding from 74 px to 12 px so the 32 px sidebar toggle occupies the space
vacated by the traffic-light controls. The position changes without animation
and returns to 74 px when leaving fullscreen. Other platforms retain their
existing titlebar geometry.

Query the native window's `isFullscreen()` state on mount and after native
resize events. Tauri does not expose a dedicated fullscreen-change window
event, so resize is the synchronization signal and `isFullscreen()` remains the
source of truth. Do not infer fullscreen from viewport dimensions. Capture
native screenshots in windowed and fullscreen states and assert the 74 px and
12 px left offsets.

### Sidebar and pane toggles

- Remove the decorative brain mark from the sidebar. Retain a compact,
  accessible folder or vault control only where an interactive trigger is
  required.
- Keep the fixed 56 px navigation gutter and existing workspace geometry.
- Set the `QUICK LINKS` heading to 12 px and link labels to 13 px, using the
  existing application typeface and design tokens.
- `Files` and `Search` use symmetric pane-toggle behaviour:
  - clicking an inactive item opens its pane;
  - clicking the active item collapses the entire Files/Search pane;
  - reopening restores the pane and its mounted state, including hierarchy
    expansion, search query, and results;
  - switching items changes the visible pane without discarding either pane's
    state.
- Persist version 2 at `nn:workspace-layout:v2` with
  `navigationExpanded`, `sidebarWidth`, and `sidebarPanel`, where
  `sidebarPanel` is `files`, `search`, or `null`.
- On first load, migrate a valid v1 `navigationExpanded` and `sidebarWidth` and
  default its panel to `files`, then write v2. Invalid v1 or v2 data falls back
  to the default layout. Do not discard a valid v1 preference.
- Keep Files and Search mounted while inactive. The inactive pane is hidden and
  inert. When `sidebarPanel` is `null`, sidebar width and splitter width are
  both 0, the titlebar grid follows that geometry, and the preferred sidebar
  width remains stored for reopening. Existing responsive editor minimums and
  navigation-collapse rules continue to apply.

### Typography and brand mark

- Render note paths in the application's primary UI typeface rather than the
  monospace face. Monospace remains appropriate for model identifiers, source
  metadata, and code.
- Replace the welcome-screen brain with the tracked NeuralNote mark at
  `assets/brand/marks/neuralnote-mark-128.png`, preserving its aspect ratio and
  accessible treatment.

## Accessibility and interaction requirements

- All new controls are keyboard operable and have visible focus states.
- Menus use the existing maintained menu primitive and expected arrow, Escape,
  Enter, and Space behaviour.
- Icon-only controls have stable accessible names and tooltips where the app
  already uses them.
- Disabled, loading, retry, stopped, dirty, saved, and error states
  are distinguishable without relying on colour alone.
- Reduced-motion behaviour follows the existing design system.
- The final real-app walkthrough includes VoiceOver checks for the model menu,
  stop action, sidebar toggle, pane toggles, source-native editor, completion,
  and save feedback.

## Security and privacy

This release changes provider traffic, untrusted Markdown parsing, webview IPC,
and persisted UI state, so it is security-sensitive.

- Keep provider secrets in the OS keychain and outside webview IPC.
- Restrict provider HTTP traffic to fixed HTTPS origins, with bounded time and
  response size.
- Validate ranking and catalogue records before sorting, joining, filtering, or
  persistence. Reject malformed, oversized, duplicate, and unsupported data
  explicitly.
- Treat Markdown, frontmatter, raw HTML, links, pasted content, and editor
  package output as untrusted.
- Do not enable script execution, unsafe URL schemes, or unsanitized HTML in
  rich or preview surfaces.
- Preserve the existing vault path, symlink, race, and atomic-write controls.
- Persist model identifiers and layout state only after validation.
- Update `NeuralNote-threat-model.md` for the new OpenRouter ranking endpoint
  and source-native decoration, completion, and navigation boundaries.
- Obtain an independent adversarial review after implementation and resolve all
  high- and critical-severity findings before the pull request is considered
  ready.

## Test-first implementation requirements

Every behaviour change starts with a failing test. At minimum, cover:

- the new Neural Assistant AI label and removal of the passive status pill;
- accessible model-menu keyboard behaviour, loading, selection, pinning,
  active-run disabling, retry, and local-settings routing;
- OpenRouter ranking and catalogue joining, latest-completed-day calculation,
  context and tool filtering, stable ordering, duplicate and malformed records,
  bounded failures, cache expiry, and no automatic model change;
- the model-only native command, generated bindings, key isolation, explicit
  errors, and mock IPC journeys;
- user-stop versus provider and lifecycle cancellation, partial-output
  retention, neutral status, and duplicate-submit prevention;
- source-native decoration, active-marker reveal, completion, exact no-op
  identity, explicit saves, conflicts, and write failures;
- full-document preservation and an adversarial corpus containing BOM, LF,
  CRLF, CR, mixed line endings, trailing spaces, YAML comments, ordering,
  anchors, aliases, multiline values, malformed frontmatter, wikilinks,
  aliases, heading and block references, embeds, callouts, block IDs, tags,
  tasks, Dataview, raw HTML, math, tables, footnotes, fenced-code metadata,
  unsafe links, and large notes;
- fullscreen event synchronization and platform-specific titlebar positions;
- sidebar semantics, typography, Files/Search toggle symmetry, mounted-state
  preservation, and layout-state migration and recovery;
- the NeuralNote welcome mark and path typography;
- accessibility regressions for labels, focus, menus, editing, save feedback,
  and error states.

Assign preservation, line-separator mapping, decoration, completion, and
state-machine requirements to pure unit tests. Use jsdom journeys for menu,
IPC, save, tab, completion, and error state, but not as proof of native editor
behaviour. Use real Tauri or browser automation for caret placement, selection,
paste, keyboard undo and redo, and link handling. Verify IME and VoiceOver
manually on macOS and record the steps and results.

Measure a 500 KiB fixture containing 5,000 supported paragraphs. On the
maintainer's release machine, median editor-ready time over five cold opens must
be at most 1.5 seconds and p95 key-to-paint latency across 20 edits must be at
most 50 ms. Lazy-load the source-native editor: the initial production
JavaScript must grow by no more than 50 KiB gzip and the editor chunk by no more
than 600 KiB gzip. Exceeding a threshold blocks the pull request unless the
maintainer explicitly accepts the measured regression.

## Verification and release preparation

During implementation, run the smallest relevant tests for each red/green
cycle. Before pushing the final branch, run and inspect:

```bash
npm --prefix app/desktop run lint
npm --prefix app/desktop run typecheck
npm --prefix app/desktop run test:unit
cargo test --workspace --locked
npm --prefix app/desktop run check:bindings
npm --prefix app/desktop run coverage
npm --prefix app/desktop run build
npm --prefix app/desktop run audit:all
./scripts/rust-quality-gate.sh
gitleaks git . --log-opts=--all --redact
```

Also complete:

- the repository's local SonarQube milestone gate. `Unavailable` must be
  reported accurately and blocks release readiness unless the maintainer
  explicitly changes the release bar;
- a real packaged-app or `tauri dev` walkthrough of every changed journey;
- an accessibility review, independent UI review, adversarial security review,
  and production-readiness audit;
- version and release-plumbing updates from 0.1.1 to 0.2.0, including package
  manifests, Cargo manifests and lockfiles, Tauri configuration, updater
  tests/harness documentation, and release workflow contract tests;
- an ad-hoc macOS package and local updater proof from 0.1.1 to 0.2.0 where the
  repository's release process permits this before publication.

Implementation takes place on an isolated feature branch. Preserve unrelated
changes in the maintainer's existing checkout. Commit coherent changes, push
the branch, then open a pull request near completion. Loop on all required
pull-request checks until they are green. Because native E2E does not run on
pull requests, manually dispatch it against the feature branch and require it
to pass before requesting merge approval.

### Approval gate 1: merge

Stop with the green pull request still open. Request explicit approval to merge
that pull request. Do not merge without it. Merge approval does not authorise a
tag, release workflow, publication, or deployment.

After an approved merge, record the exact `main` commit and loop on main CI and
native E2E until both are green.

### Approval gate 2: release

Only after the merged commit is green may release approval be requested. The
request and approval must identify:

- tag `v0.2.0`;
- the exact green `main` commit;
- ad-hoc signing; and
- acknowledgement that the macOS build is unnotarized.

Do not create or push `v0.2.0`, dispatch the release workflow, publish assets,
or deploy anything before this second approval.

### Post-publication acceptance

After an approved dispatch, the release is complete only when all of these are
verified:

- the release is immutable, marked as a prerelease, and is not marked latest;
- it contains the expected DMG, updater archive, updater signature, and updater
  manifest, with no unexpected release assets;
- the manifest version is 0.2.0 and includes the `darwin-aarch64` target;
- the updater signature is non-empty and the archive URL uses HTTPS;
- the published raw updater manifest matches the release manifest;
- a retained or installed 0.1.1 build detects 0.2.0, installs it, relaunches,
  and reports failures visibly; and
- the final release URL, tag, commit, workflow conclusion, signing mode, and
  verification evidence are recorded in the handoff.

If publication is incomplete or inconsistent, do not move or recreate the
immutable tag or silently replace assets. Stop and follow the documented
release recovery procedure with maintainer approval.

## Non-goals

- Merging the pull request without explicit maintainer approval.
- Creating a release tag or publishing/deploying 0.2.0 before approval gate 2.
- Replacing the existing local-model settings and lifecycle workflow.
- Automatically choosing a newly ranked OpenRouter model.
- Changing the public `ChatEvent` wire contract.
- Claiming full-source ingestion, chunk-level citations, or timestamp-level
  recall.
- Rendering every possible Obsidian extension in live preview during this
  release; unsupported syntax must still remain source-editable.
- Sacrificing source preservation to make a note appear rendered.
