# Contributing to NeuralNote

Thanks for helping with NeuralNote. This project handles user-owned files and AI-generated claims, so a small, well-tested change is more useful than a broad change that is difficult to verify.

## Before you start

1. Search existing issues and pull requests.
2. Open an issue before changing architecture, data formats, provider behaviour, citation contracts, or user-facing workflows.
3. Read the [product specification](specs/neural-note.md) and [Definition of Done](docs/definition-of-done.md).
4. If you are using a coding agent, give it [AGENTS.md](AGENTS.md) as project guidance.

Small fixes, tests, and documentation corrections can go straight to a focused pull request when the intent is clear.

## Set up the repository

The project uses Node.js 22, Rust 1.96, React 19, and Tauri 2. Install the [Tauri system prerequisites](https://v2.tauri.app/start/prerequisites/) for your operating system first.

```bash
git clone https://github.com/ThomasPritchard/NeuralNote.git
cd NeuralNote

rustup toolchain install 1.96.0 --component clippy,rustfmt,llvm-tools-preview
npm --prefix app/desktop ci
```

For the native app on macOS, fetch the checksum-pinned Ollama sidecar:

```bash
./scripts/fetch-ollama-sidecar.sh
npm --prefix app/desktop run tauri dev
```

For frontend-only work:

```bash
npm --prefix app/desktop run dev
```

## Understand the boundaries

- `crates/neuralnote-core/` owns reusable product logic: vault I/O, parsing, search, links, retrieval, citation evidence, and provider-independent orchestration.
- `app/desktop/src-tauri/` is the thin native shell: Tauri commands, keychain access, networking, process management, and desktop integration.
- `app/desktop/src/` is the React webview. Native calls go through `src/lib/api.ts`.
- `app/desktop/src/lib/bindings/` is generated by `ts-rs`. Never edit it manually.
- `prototype/` is design exploration, not shipping product code or a pattern library.

Keep Markdown and YAML frontmatter compatible with ordinary Obsidian vaults. Do not introduce a proprietary data dependency into the local file format.

## Make the change

- Add a failing test before implementing a feature or bug fix.
- Cover the successful path and relevant malformed, empty, oversized, non-UTF-8, cancellation, and concurrency cases.
- Put reusable logic in `neuralnote-core`; keep Tauri commands focused on native I/O and delegation.
- Surface failures to the user. Do not silently drop content or hide parse, provider, capture, or citation failures.
- Treat vault content, model responses, helper output, filenames, and IPC arguments as untrusted.
- Preserve citation provenance. Generated citation markers are not trusted evidence by themselves.
- Check current primary documentation before adding or changing an external library, API, CLI, or cloud service.
- Avoid unrelated formatting or refactors in the same pull request.

When a Rust type crosses the Tauri boundary, regenerate its TypeScript mirror:

```bash
npm --prefix app/desktop run gen:bindings
```

## Verify the change

Run the checks that cover the code you touched. A normal pull request should run at least:

```bash
npm --prefix app/desktop run lint
npm --prefix app/desktop run typecheck
npm --prefix app/desktop run test:unit
cargo test --workspace --locked
npm --prefix app/desktop run check:bindings
```

Pull requests targeting `main` run these fast, secret-free checks in GitHub Actions. Native WebDriver tests and the complete coverage, build, and dependency-audit suite run after merge on pushes to `main`.

Install the Rust gate tools once:

```bash
cargo install cargo-llvm-cov --locked --version 0.8.7
cargo install cargo-audit --locked --version 0.22.2
```

Then run the project gate:

```bash
./scripts/rust-quality-gate.sh
```

Maintainers also use the [local Docker SonarQube gate](docs/local-sonarqube.md) for milestones and release readiness. It is optional for external contributors. Do not request or share a maintainer's analysis token.

User-facing work also needs a real-app check with `npm --prefix app/desktop run tauri dev`. Record what you exercised in the pull request.

Changes involving parsers, untrusted content, filesystem paths, IPC, secrets, provider traffic, downloaded tools, or authentication need an independent adversarial review. Green tests and static analysis are not sufficient for those boundaries. The complete acceptance criteria live in [docs/definition-of-done.md](docs/definition-of-done.md).

## Open a pull request

Keep the pull request easy to review:

- Explain the user problem and the chosen scope.
- Link the issue or specification section that defines the behaviour.
- List the files or architectural boundaries changed.
- Include the exact verification commands and their results.
- Add screenshots or a short recording for visual changes.
- Call out security, privacy, data migration, compatibility, and citation implications.
- State any skipped check and why it could not run.
- Do not combine generated output, dependency upgrades, and unrelated cleanup unless they are required by the same change.

Target pull requests at `main`. Maintainers configure branch protection to require the fast `Secrets / Gitleaks`, `Frontend / lint, types, and tests`, and `Rust / test, lint, format, and bindings` checks.

Maintainers may ask for a smaller pull request when a change mixes concerns or makes citation and vault-safety review difficult.

## Commit and repository hygiene

Use short imperative commit subjects. Keep commits coherent enough to review or revert.

Never commit:

- `.env` files, tokens, credentials, certificates, or signing material
- local vaults or user content
- `target/`, `node_modules/`, coverage reports, Sonar scanner output, or local logs
- downloaded Ollama or helper binaries
- pasted root-level screenshots or production-audit working reports

Run a secret scan before publishing a branch when configuration or CI changed:

```bash
gitleaks git . --log-opts=--all --redact
```

If a real secret ever enters Git history, removing the file is not enough. Revoke or rotate the credential before rewriting or cleaning the history.

## Security reports

Do not open a public issue for a suspected vulnerability or include exploit details in a public pull request. Follow [SECURITY.md](SECURITY.md), which defines the private reporting route, supported versions, and disclosure process.

## AI-assisted contributions

AI-assisted work is welcome, but the contributor remains responsible for every line, test result, dependency choice, and claim in the pull request. Review generated code, remove fabricated references, and disclose substantial agent-generated changes in the pull request description so reviewers know where extra scrutiny may help.
